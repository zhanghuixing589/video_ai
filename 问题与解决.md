## 1. 前端请求地址与后端接口不匹配

- **日期**：2026-06-12
- **现象**：前端请求 `/api/auth/login` 时无法正确访问后端登录接口。
- **影响**：登录、注册以及其他 API 请求失败。
- **根因**：后端配置了 `/api` context path，但 Vite 代理又将 `/api` 前缀删除，最终请求路径错误。
- **解决方案**：保留 Vite 代理中的 `/api` 前缀，只将请求转发到 `http://localhost:8080`。
- **验证方式**：浏览器访问登录页并调用登录接口；公开内容接口返回 HTTP 200。
- **后续改进**：使用环境变量统一管理不同环境的 API 地址。

## 2. 前端错误解析统一响应结构

- **日期**：2026-06-12
- **现象**：登录成功后读取不到 `token` 和用户信息。
- **影响**：无法保存登录状态，也无法按照用户角色跳转。
- **根因**：后端响应结构是 `{ success, message, data }`，前端却直接把外层响应当作业务数据。
- **解决方案**：在 Axios API 层统一解包 `response.data.data`，页面只接收业务对象。
- **验证方式**：管理员登录后可获得 JWT，并自动进入 `/admin`。
- **后续改进**：根据 OpenAPI 自动生成前端接口类型，减少前后端契约不一致。

## 3. TypeScript 配置与编译器版本不兼容

- **日期**：2026-06-12
- **现象**：执行 `npm run build` 时出现 `erasableSyntaxOnly` 未知配置，以及 `tsBuildInfoFile` 缺少增量编译配置。
- **影响**：前端无法构建。
- **根因**：脚手架生成的配置项高于项目锁定的 TypeScript 5.5 能力。
- **解决方案**：删除不支持的 `erasableSyntaxOnly` 和不必要的 `tsBuildInfoFile` 配置。
- **验证方式**：`npm run build` 成功完成。
- **后续改进**：升级依赖时同时检查 TypeScript、Vite 和脚手架配置版本。

## 4. 登录页面角色选择器无实际作用

- **日期**：2026-06-12
- **现象**：登录页面允许选择普通用户、制片厂、审核员和管理员，但该选择没有发送给后端。
- **影响**：容易让用户误以为可以自行选择身份登录。
- **根因**：用户角色应由数据库决定，前端角色选择只是一项没有业务效果的界面状态。
- **解决方案**：移除登录角色选择器，登录后按照服务端返回的角色和制片厂状态自动分流。
- **验证方式**：管理员、审核员、普通用户和制片厂使用同一登录页，并进入各自页面。
- **后续改进**：所有权限判断必须由后端执行，前端只负责展示和导航。

## 5. 注册接口存在但前端没有注册入口

- **日期**：2026-06-12
- **现象**：后端已有 `/auth/register`，但用户无法从页面注册。
- **影响**：普通用户和制片厂账号无法自行创建。
- **根因**：前后端功能没有完整接通。
- **解决方案**：将登录与注册整合在统一认证页；公开注册只允许 `USER` 和 `STUDIO`，注册成功后切换回登录模式。
- **验证方式**：注册表单可选择普通用户或制片厂，并显示“注册成功，请登录”。
- **后续改进**：增加邮箱验证、密码强度提示和找回密码功能。

## 6. `/auth/me` 被公开放行

- **日期**：2026-06-12
- **现象**：安全配置使用 `/auth/**` 全部放行，导致当前用户接口也不要求认证。
- **影响**：认证边界不清晰，容易出现无效 token 被错误处理的问题。
- **根因**：公开接口匹配范围过大。
- **解决方案**：只放行 `/auth/login` 和 `/auth/register`，`/auth/me` 必须经过 JWT 认证。
- **验证方式**：匿名访问 `/api/auth/me` 返回 HTTP 401。
- **后续改进**：为所有敏感接口补充权限集成测试。

## 7. 全局异常处理缺失

- **日期**：2026-06-12
- **现象**：业务异常可能返回 500 或不同格式的错误响应。
- **影响**：前端无法稳定展示错误信息，也无法区分 400、401、403 和 409。
- **根因**：原全局异常处理类为空，业务代码大量抛出通用异常。
- **解决方案**：增加 `ApiException` 和全局异常处理器，统一返回 `ApiResponse`；参数错误、认证失败、权限不足和数据冲突使用正确状态码。
- **验证方式**：相关单元测试通过，未登录访问返回 JSON 401。
- **后续改进**：增加错误编码、请求 ID 和结构化错误日志。

## 8. 受保护路由只信任 localStorage

- **日期**：2026-06-12
- **现象**：只要本地存储存在 token 和用户 JSON，前端就允许进入受保护路由。
- **影响**：过期或伪造的本地数据可能暂时显示无权限页面。
- **根因**：路由守卫没有向服务端验证 token。
- **解决方案**：受保护路由进入时调用 `/auth/me`，根据服务端返回的真实角色和状态决定跳转。
- **验证方式**：未登录访问 `/studio` 会跳转到 `/login`。
- **后续改进**：引入统一认证上下文，减少页面重复读取 localStorage。

## 9. 制片厂注册与申请流程混在一起

- **日期**：2026-06-12
- **现象**：制片厂可能被要求注册时立即提交完整资料。
- **影响**：不符合制片厂可以稍后准备片库和申请材料的业务需求。
- **根因**：账号注册和资质申请没有拆分。
- **解决方案**：制片厂注册后状态为 `NONE`；登录后进入申请页，可稍后提交；`PENDING` 等待审核，`REJECTED` 可修改后重新提交，`APPROVED` 才能进入工作台。
- **验证方式**：制片厂申请状态单元测试通过。
- **后续改进**：记录审核意见、申请历史和审核操作日志。

## 10. 单条 Video 模型无法管理电视剧和综艺

- **日期**：2026-06-12
- **现象**：电视剧和综艺的每一集都需要重复填写整部作品的信息。
- **影响**：数据重复，季度和剧集难以排序、更新和审核。
- **根因**：原模型只有单层 `Video`。
- **解决方案**：新增 `Content`、`Season` 和 `Episode` 模型；电影使用一个无季度剧集，电视剧和综艺通过季度组织多集。
- **验证方式**：作品、季度和剧集服务测试通过，制片厂工作台可分别创建。
- **后续改进**：增加批量导入、文件上传、转码状态和旧视频数据迁移。

## 11. 游客试看缺少业务规则

- **日期**：2026-06-12
- **现象**：公开首页与完整视频观看没有区分。
- **影响**：游客访问策略不明确。
- **根因**：播放器没有根据认证状态限制播放时长。
- **解决方案**：每个剧集默认 `previewSeconds=300`；游客播放到五分钟时自动暂停并提示登录或注册。
- **验证方式**：公开首页可匿名访问，播放器组件按剧集试看时间执行限制。
- **后续改进**：视频源改为签名 HLS 地址，避免用户直接绕过前端访问公开 URL。

## 12. 缺少默认管理员账号

- **日期**：2026-06-12
- **现象**：系统启动后没有可用于进入管理后台的账号。
- **影响**：无法审核制片厂、管理用户和创建审核员。
- **根因**：公开注册禁止管理员，但没有开发环境初始化账号。
- **解决方案**：增加仅在 `dev` profile 生效的 `DevAdminInitializer`，管理员不存在时使用 BCrypt 创建默认账号，存在时不覆盖。
- **验证方式**：使用 `admin / Admin@123456` 调用真实登录接口成功。
- **后续改进**：生产环境使用部署密钥或一次性初始化流程，不使用固定默认密码。

## 13. 管理员控制台不能创建审核员

- **日期**：2026-06-12
- **现象**：后端存在用户管理接口，但管理员控制台没有创建账号和调整角色的入口。
- **影响**：审核员账号无法正常获得。
- **根因**：管理业务只完成了后端部分，没有完整前端流程。
- **解决方案**：管理员控制台增加创建账号弹窗和角色调整控件；支持创建普通用户、制片厂和审核员。禁止直接创建管理员，禁止管理员降级自己，并保证至少保留一个管理员。
- **验证方式**：后端角色规则测试通过；浏览器中可看到创建审核员表单和角色管理表格。
- **后续改进**：增加禁用账号、重置初始密码和操作审计。

## 14. 端口 8080 被重复启动的后端占用

- **日期**：2026-06-12
- **现象**：IDEA 启动时报错 `Port 8080 was already in use`。
- **影响**：新的 Spring Boot 进程无法启动。
- **根因**：功能验证时后台启动的 `VideoAiApplication` 仍在监听 8080。
- **解决方案**：通过 `netstat` 获取 PID，核对进程命令行确认属于当前项目后停止该进程。
- **验证方式**：再次检查 8080，无监听进程。
- **后续改进**：验证结束后及时停止后台服务；启动失败时先检查端口占用，避免直接结束不明进程。

## 15. 项目仍以 CRUD 为主，缺少生产级视频能力

- **日期**：2026-06-12
- **现象**：目前没有视频转码、高并发播放、防盗链、搜索引擎、单设备登录和完整互动数据链路。
- **影响**：项目业务复杂度、系统设计和生产就绪程度仍然有限。
- **根因**：当前阶段优先完成了认证、角色、内容建模和基础业务闭环。
- **解决思路**：
  - 使用 MinIO/S3 保存原始视频和转码结果。
  - 使用 FFmpeg 将视频转为多码率 HLS。
  - 使用消息队列异步执行转码、重试和搜索索引更新。
  - 使用 CDN 与签名 URL 分发视频并防盗链。
  - 使用 Redis 管理单设备会话、限流、热门缓存、播放进度缓冲和点赞计数。
  - 使用独立的播放进度、点赞、收藏和评论表，唯一索引保证幂等。
  - 小规模搜索使用 PostgreSQL 全文检索和 `pg_trgm`，数据增长后迁移到 Elasticsearch/OpenSearch。
  - 使用 Actuator、Prometheus 和 Grafana 建立监控。
- **验证方式**：后续每个阶段分别增加性能测试、故障测试和接口集成测试。
- **后续改进**：建议下一阶段优先实现 Redis 单设备登录、播放进度、点赞评论、限流和监控，再实现转码与对象存储。

## 16. AdminDashboard 和 Login 被 ESLint 配置错误连带标红

- **日期**：2026-06-12
- **现象**：`AdminDashboard.tsx` 和 `Login.tsx` 在编辑器中报错，执行 `npm run lint` 时 ESLint 在加载配置阶段直接崩溃。
- **影响**：无法获得页面代码的真实静态检查结果，页面中的有效错误被配置异常遮挡。
- **根因**：项目安装的 `eslint-plugin-react-hooks` 提供 `recommended-latest` 配置，但 `eslint.config.js` 错误引用了不存在的 `reactHooks.configs.flat.recommended`；配置修复后，两个页面还各有一处丢弃 `confirmPassword` 时产生的未使用变量。
- **解决方案**：将 Hooks 配置改为 `reactHooks.configs['recommended-latest']`；登录注册和管理员创建账号时显式构造 API 请求对象，不再通过未使用的解构变量移除确认密码。
- **验证方式**：两个页面的定向 ESLint 检查通过；`npm run build` 成功；登录与注册界面在浏览器中正常切换且控制台无错误。
- **后续改进**：继续处理其他页面现有的 `no-explicit-any` 和 Hooks 依赖警告，使全项目 `npm run lint` 完整通过。

## 18. 退出后首页仍显示上一次登录用户

- **日期**：2026-06-12
- **现象**：审核员张三退出并回到登录页后，重新启动程序访问 `localhost:3000`，首页仍显示“欢迎，张三”。
- **影响**：游客会被错误展示为已登录状态，试看限制也会因为缓存用户而被绕过。
- **根因**：`localStorage` 会跨浏览器和程序重启持久保存；首页只读取缓存中的 `user`，没有检查 `token` 是否存在，也没有调用 `/auth/me` 验证登录状态。
- **解决方案**：新增统一认证会话模块。首页初始化时，没有 token 就清除残留用户；有 token 则调用 `/auth/me`，只使用服务端验证后的用户；验证失败时清空 token 和用户缓存。Axios 的 401 清理和主动退出也统一使用同一清理函数。
- **验证方式**：增加 3 个认证会话回归测试，覆盖无 token 残留用户、有效 token 和校验失败场景；定向 ESLint 与生产构建通过。
- **后续改进**：后续可将用户状态提升到统一的 React Auth Context，并通过 `storage` 事件同步多个浏览器标签页的登录和退出状态。

## 19. 前端可运行但 IDE 和 ESLint 持续爆红

- **日期**：2026-06-12
- **现象**：开发服务器可以正常运行，但 `StudioReviewerDashboard.tsx` 存在 6 个 `any` 错误和 Hook 依赖警告，`StudioWorkspace.tsx` 也有 Hook 依赖警告。
- **影响**：开发模式没有阻止页面启动，但缺失类型会掩盖接口结构变化，Hook 闭包可能读取旧状态，IDE 持续显示错误。
- **根因**：旧视频 API 没有声明请求和响应类型，页面直接处理 Axios 原始响应并大量使用 `any`；两个异步加载函数每次渲染都会创建新引用，却没有正确声明 Hook 依赖。
- **解决方案**：补充视频、创建视频和审核视频的 TypeScript 类型；API 层统一解包业务数据；审核员页面使用 `UserInfo` 和 `/auth/me`；表格回调使用 `unknown`；异步加载函数使用 `useCallback` 并补全依赖。
- **验证方式**：执行 `npm run lint` 无错误和警告；前端测试与生产构建通过。
- **后续改进**：逐步将后端 DTO 通过 OpenAPI 自动生成前端类型，避免接口结构变化后各页面手工同步。

## 20. 单设备登录失效后跳转登录页但没有提示

- **日期**：2026-06-12
- **现象**：账号在其他设备重新登录后，旧设备刷新会正确跳转到登录页，但看不到“您的账号已在其他设备登录，请重新登录”。
- **影响**：用户不知道自己为什么退出，容易误以为登录过期或系统故障。
- **根因**：最初拦截器在整页跳转前显示消息，页面卸载导致消息消失；改为 `sessionStorage` 后仍未生效，是因为 `JwtAuthenticationFilter` 没有设置 UTF-8，真实响应变成 `{"message":"??????????????????"}`，前端无法识别中文。另外，React 开发模式的 `StrictMode` 会重复执行挂载流程，“读取即删除”可能在第一次模拟挂载时提前消费提示。
- **解决方案**：过滤器明确返回 UTF-8，并增加稳定响应头 `X-Auth-Reason: SESSION_REPLACED`；前端优先按原因代码识别，中文文本仅作兼容回退。登录页改为先读取但不立即删除，使用固定消息 key 显示，并在挂载稳定后延迟清除提示代码。
- **验证方式**：后端过滤器测试验证 401、UTF-8、原因代码和完整中文；前端认证提示测试覆盖保存、读取、清除、单次消费和原因识别；真实执行两个客户端依次登录同一管理员账号后，旧客户端刷新跳转 `/login`，浏览器检测到目标提示恰好一条。
- **后续改进**：可将认证失效原因统一定义为后端错误码，避免前端依赖中文消息文本进行分类。

## 21. 同一浏览器中的两个客户端会互相覆盖和注销会话

- **日期**：2026-06-12
- **现象**：第一个客户端登录管理员后，第二个客户端登录同一账号会收到“账号已在其他设备登录”；第一个客户端退出后，第二个客户端仍无法继续使用。
- **影响**：新登录客户端会被旧客户端的请求或退出操作误伤，不符合“新登录会话有效、旧会话失效”的单设备登录规则。
- **根因**：前端把 JWT 放在同源共享的 `localStorage`。第二个标签页登录后会覆盖第一个标签页读取到的 token，第一个标签页退出时实际提交的是第二个标签页的新 token。并且旧请求稍后返回的 401 会无条件清空刚写入的新 token。后端退出接口也只按 `userId` 删除 Redis 会话，缺少 `sessionId` 匹配保护。
- **解决方案**：认证 token 和用户信息统一改存每个标签页独立的 `sessionStorage`；Axios 收到 401 时比较该请求携带的 token 与当前 token，旧请求响应不得清理新会话；退出接口从 JWT 提取 `userId` 与 `sessionId`，仅在 Redis 当前 session 匹配时删除。
- **验证方式**：前端回归测试覆盖旧 token 的 401 不清理新 token；后端回归测试覆盖旧 session 不能删除当前 session、当前 session 可以正常退出；使用两个 token 依次登录同一账号，验证旧 token 返回 401、新 token 返回 200、旧 token 退出不影响新 token。
- **后续改进**：将 Redis 的“比较 sessionId 后删除”升级为 Lua 脚本原子操作，并增加真实 Redis 的并发集成测试。

## 27. 发布接口返回 PUBLISHED 但数据库作品状态仍为 PENDING

- **日期**：2026-06-14
- **现象**：作品重新审核为 `APPROVED` 后，`PATCH /api/contents/2/publish` 返回 HTTP 200，响应 DTO 的状态也是 `PUBLISHED`，剧集 `published_at` 已写入；但随后直接查询 `contents` 表，作品状态仍为 `PENDING`，公开内容接口也查不到该作品。
- **影响**：前端会提示“作品已发布”，实际数据库却没有完成作品状态迁移，造成内存状态、接口响应和持久化状态互相矛盾。
- **排查证据**：发布事务日志显示同一请求执行了 `UPDATE contents` 和 `UPDATE episodes`，响应对象为 `PUBLISHED`；数据库行的 `updated_at` 与该事务时间一致，但状态仍错误。没有后续请求、数据库触发器或状态重置逻辑能够解释这次回退，因此必须在发布方法内部把作品状态的落库作为明确的事务边界验证。
- **根因**：发布方法设置实体状态后只调用普通 `contentRepository.save(content)`，随即继续组装包含关联查询的 DTO。该路径只证明托管实体在内存中已变为 `PUBLISHED`，没有为关键状态迁移建立立即 flush 的持久化边界，因而接口成功响应无法作为数据库已更新的证据。
- **解决方案**：发布方法在设置 `content.status=PUBLISHED` 后改用 `contentRepository.saveAndFlush(content)`，使用其返回实体生成 DTO；剧集发布时间与作品状态仍处于同一事务中，任一落库失败都会使请求失败并回滚。新增回归测试明确验证发布路径调用 `saveAndFlush`。
- **验证方式**：修复后使用当前审核员真实会话依次调用审核和发布接口，响应状态分别为 `APPROVED`、`PUBLISHED`；随后独立查询 PostgreSQL，确认 `contents.id=2` 为 `PUBLISHED`、对应剧集 `published_at` 已刷新；匿名调用 `/api/contents/public` 可以查到该作品。前端 24 个测试、定向 ESLint 和生产构建通过，后端发布回归测试通过。
- **后续改进**：涉及审核、发布、支付等关键状态迁移时，测试不能只断言返回 DTO，必须同时约束持久化调用，并优先增加使用隔离测试数据库的仓储或接口集成测试。端到端验收要同时检查 HTTP 响应、数据库最终状态和下游查询结果。
