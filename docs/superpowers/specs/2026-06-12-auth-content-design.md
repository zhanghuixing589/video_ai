# 认证、游客试看与分集内容设计

## 目标

修复现有登录链路，补全普通用户和制片厂注册，允许游客访问公开首页并试看每集前五分钟，同时将当前单条视频模型升级为适合电影、电视剧和综艺的作品、季度、剧集结构。

## 用户流程

### 游客

- `/` 是公开页面，无需 JWT。
- 游客可以浏览已发布作品、季度和剧集信息。
- 游客播放任一剧集时只能观看前 300 秒。
- 播放位置到达 300 秒时播放器暂停，并显示登录和注册入口。
- 试看限制按剧集计算。登录用户观看不受该前端限制。
- 首版视频仍使用 URL。平台播放器能执行试看限制，但不能阻止用户绕过平台直接访问公开源地址；生产环境应使用受控对象存储和签名播放地址。

### 注册

- 登录页提供“登录”和“注册”两个模式。
- 公开注册只允许 `USER` 和 `STUDIO`。
- `USER` 注册为 `role=USER`、`studioStatus=NONE`。
- `STUDIO` 注册为 `role=STUDIO`、`studioStatus=NONE`，注册阶段不强制填写制片厂资料。
- `ADMIN` 和 `REVIEWER` 不允许公开注册，只能由受保护的后台能力创建或调整。
- 两类公开注册成功后都切换到登录模式，并提示“注册成功，请登录”，不自动签发 JWT。

### 登录

- 用户可以使用用户名或邮箱登录。
- 前端不再要求用户预选角色；角色以服务端返回的数据为准。
- 登录成功后保存 JWT 和用户概要。
- `USER` 返回公开首页。
- `STUDIO + NONE` 进入制片厂申请页。
- `STUDIO + PENDING` 进入申请进度页。
- `STUDIO + REJECTED` 进入申请页，展示拒绝状态并允许修改后重新提交。
- `STUDIO + APPROVED` 进入制片工作台。
- `REVIEWER` 进入审核工作台，`ADMIN` 进入管理后台。
- 禁用账号、错误凭据和无效 JWT 返回一致、可展示的错误响应。

## 认证与授权

- `/auth/login`、`/auth/register`、公开作品查询接口允许匿名访问。
- `/auth/me` 必须经过 JWT 过滤器认证，不再自行解析可选请求头。
- JWT 权限使用 `ROLE_` 前缀，角色和制片厂状态均由数据库状态决定。
- 服务端通过方法或请求级授权保护管理、审核、制片内容写入接口。
- 制片厂申请接口只允许当前登录的制片厂操作自己的账号，用户 ID 从认证主体获取，不信任请求体或路径提供的任意身份。
- 只有 `STUDIO + APPROVED` 可以创建、修改和提交作品或剧集。
- `ADMIN` 才能管理用户角色；`ADMIN` 或 `REVIEWER` 才能审核内容。

## API 响应与校验

- 所有 API 保持 `{ success, message, data }` 结构。
- 前端统一从 `response.data.data` 读取业务数据。
- 参数校验错误返回 HTTP 400 和第一条可读字段错误。
- 用户名或邮箱冲突返回 HTTP 409。
- 登录失败返回 HTTP 401，禁用账号返回 HTTP 403。
- 未认证访问返回 HTTP 401，权限不足返回 HTTP 403，并使用 JSON 响应。
- 用户名和邮箱在写入前去除首尾空白；邮箱以小写存储和查询。
- 密码使用 BCrypt 存储，公开响应永不包含密码哈希。
- 修复现有中文源码和提示文本的乱码。

## 前端路由

- `/`：公开作品首页。
- `/login`：统一登录/注册页。
- `/studio/application`：制片厂申请和进度页。
- `/studio`：审核通过后的制片工作台。
- `/reviewer`：审核工作台。
- `/admin`：管理员后台。
- 受保护路由启动时调用 `/auth/me` 校验 token，不能只信任 `localStorage` 中的用户 JSON。
- 401 响应清除本地认证状态并跳转登录；公开首页请求出现 401 时不应形成跳转循环。

## 制片厂申请

- 申请字段包含制片厂名称和简介。
- 首次提交将状态从 `NONE` 变为 `PENDING`。
- 管理员审核结果只能为 `APPROVED` 或 `REJECTED`。
- 被拒绝后制片厂可修改资料并重新提交，状态重新变为 `PENDING`。
- 未申请、待审核和被拒绝状态均不能创建内容。

## 内容数据模型

### Content

表示一部作品，包含：

- `id`
- `title`
- `description`
- `coverUrl`
- `type`：电影、电视剧、综艺
- `genre`
- `status`
- `studioId`
- 审核人、审核时间、审核意见
- 创建和更新时间

### Season

表示作品的一季，包含：

- `id`
- `contentId`
- `seasonNumber`
- `title`
- `sortOrder`

电视剧和综艺通过季度组织剧集。电影不需要季度。

### Episode

表示实际播放单元，包含：

- `id`
- `contentId`
- 可选的 `seasonId`
- `episodeNumber`
- `title`
- `videoUrl`
- `durationSeconds`
- `previewSeconds`，默认 300
- `sortOrder`
- `publishedAt`
- 创建和更新时间

电影创建一个不属于季度的剧集。电视剧和综艺的剧集必须属于对应季度。

## 内容管理

- 制片厂可以创建作品，随后添加季度和剧集。
- 剧集支持单条添加和一次提交多条 URL 的批量创建。
- 批量接口逐项校验集数、标题、URL 和时长；同一季度内集数不可重复。
- 首版不实现本地视频上传，仅管理视频 URL。
- 作品是主要审核单位。新增或替换已审核作品中的剧集后，作品重新进入待审核状态。
- 仅已发布作品和可发布剧集出现在公开接口中。

## 现有数据迁移

- 新增 `contents`、`seasons` 和 `episodes` 表。
- 保留旧 `videos` 表，避免 Hibernate 自动更新造成破坏性迁移。
- 提供显式迁移逻辑或 SQL，将每条旧视频转换为一个作品和一个剧集。
- 在迁移验证完成前不删除旧实体或表。

## 已确认缺陷

- Vite 代理当前删除 `/api`，与后端 `/api` context path 冲突。
- 登录前端把统一响应外层误当成登录数据。
- 登录角色选择器不参与请求或校验，造成误导。
- 后端注册接口存在，但前端没有注册入口。
- `/auth/**` 全部公开导致 `/auth/me` 也未受保护。
- 全局异常处理类为空，错误状态和响应体不一致。
- 受保护路由只信任本地存储，不能验证 token 是否过期或被篡改。
- 当前 TypeScript 5.5 不支持脚手架中的 `erasableSyntaxOnly`，且 `tsBuildInfoFile` 缺少 `incremental` 或 `composite`。

## 测试策略

- `AuthService` 单元测试覆盖用户名或邮箱登录、密码错误、禁用账号、普通用户注册、制片厂注册、重复用户名和邮箱标准化。
- MVC 集成测试覆盖注册校验、登录响应、401、403 和统一异常结构。
- Security 测试验证公开首页接口、受保护的 `/auth/me`、角色权限和制片厂状态权限。
- 制片厂申请测试覆盖首次提交、重复提交、拒绝后重提和他人身份隔离。
- 内容服务测试覆盖电影单集、电视剧季度剧集、重复集数、批量创建以及内容状态重置。
- 前端测试覆盖统一认证页、注册成功切回登录、角色分流、公开首页和五分钟试看弹窗。
- 最终验证包含 Maven 测试、前端测试、TypeScript 构建和浏览器端完整流程。

## 非目标

- 本次不实现视频文件上传、转码、DRM、支付订阅、短信或邮件验证。
- 本次不提供公开管理员或审片员注册。
- 本次不删除旧视频表。
